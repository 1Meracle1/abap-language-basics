# Releasing

This repository publishes one platform-neutral VSIX under the extension ID
`1meracle1.abap-language-basics`. A matching `vX.Y.Z` tag drives two independent
release systems:

| System | Definition | Output |
| --- | --- | --- |
| GitHub Actions | `.github/workflows/release.yml` | GitHub Release and Open VSX |
| Azure Pipelines | `azure-pipelines.yml` | Visual Studio Marketplace |

Both systems install the locked dependencies, require the tag to equal
`v${package.json.version}`, run the VS Code-host tests, build the VSIX, and
verify its contents before publishing. GitHub Actions builds once and passes the
same artifact to its GitHub Release and Open VSX jobs. Azure Pipelines rebuilds
the package independently before Marketplace publication.

## One-time configuration

### GitHub and Open VSX

1. Claim the `1meracle1` namespace on Open VSX and complete any required account
   agreement.
2. Create an Open VSX publishing token for that namespace.
3. Create a protected GitHub environment named `release` and add the token as
   the `OVSX_PAT` environment secret.
4. Keep GitHub Actions permissions restricted to those declared by the
   workflows. Only the `github-release` job needs `contents: write`.

### Visual Studio Marketplace

The Marketplace pipeline uses Microsoft Entra workload identity federation;
it must not depend on a personal or organization-wide Azure DevOps PAT.

1. Create a user-assigned managed identity named
   `abap-language-basics-marketplace`.
2. Create a manual Azure Resource Manager workload-identity service connection
   with the same name in the dedicated Azure DevOps project. Complete the
   federated credential on the managed identity using the issuer and subject
   generated by Azure DevOps.
3. Add the managed identity's service principal to the Azure DevOps
   organization. This provisions the Azure DevOps profile used by Marketplace.
   Project membership is not required.
4. Resolve the identity's **Azure DevOps profile resource ID** while
   authenticated through the workload-identity service connection:

   ```sh
   az rest \
     --url https://app.vssps.visualstudio.com/_apis/profile/profiles/me \
     --resource 499b84ac-1321-427f-aa17-267ca6975798 \
     --query id \
     --output tsv
   ```

5. In the Visual Studio Marketplace publisher `1meracle1`, add that resource ID
   as a member with the **Contributor** role.
6. Create an Azure Pipeline connected to this repository and select
   `azure-pipelines.yml`. Authorize the service connection for the pipeline.

The Marketplace member identifier is the GUID returned by the profile API. It
is not the managed identity's Azure client ID, object/principal ID, subscription
resource ID, or Azure role assignment. If the profile API returns
`VSS011031: There is no profile for the authenticated user`, add the service
principal to the Azure DevOps organization and retry after provisioning.

The Marketplace Contributor role and Azure RBAC are separate controls. Azure
RBAC lets the service connection authenticate; the publisher membership lets
`vsce` publish the extension.

## Release procedure

1. Update `package.json` and `package-lock.json` to the same SemVer version.
2. Update `CHANGELOG.md` with the user-visible changes.
3. Run the complete local validation:

   ```sh
   npm ci
   npm test
   npm run verify:package
   ```

4. Merge the release commit to `main` and wait for `.github/workflows/ci.yml` to
   pass.
5. Create the immutable matching tag and push it:

   ```sh
   git tag vX.Y.Z
   git push origin vX.Y.Z
   ```

6. Confirm the GitHub Actions `Release` workflow and the Azure Pipeline both
   selected the tag, not `main` or another branch.
7. Verify all three public results:

   - GitHub: `https://github.com/1Meracle1/abap-language-basics/releases/tag/vX.Y.Z`
   - Open VSX: `https://open-vsx.org/extension/1meracle1/abap-language-basics`
   - Marketplace: `https://marketplace.visualstudio.com/items?itemName=1meracle1.abap-language-basics`

8. Update the `abap-lsp` submodule pointer to the released commit only after the
   standalone extension is available in both registries. Release any ABAP LSP
   package that declares this extension in `extensionDependencies` afterward.

The `v0.0.1` Azure run was started manually against `refs/tags/v0.0.1` because
the Azure Pipeline was created after that tag already existed. Future new `v*`
tags are covered by the pipeline trigger and should not need a manual run.

## Checks and invariants

- `scripts/check-version.mjs` rejects a tag that does not exactly match the
  package version. Do not bypass this check or move an existing release tag.
- `npm ci` enforces the committed lockfile and pinned publisher tools.
- `npm test` launches a clean VS Code Electron host and checks language
  registration, `.abap` recognition, snippets, and folding ranges.
- `npm run verify:package` creates the VSIX and enforces the runtime allowlist.
  A release package must contain only the manifest, license, README, changelog,
  grammar, snippets, and language configuration.
- The VSIX is platform-neutral. Do not add a platform target unless the
  extension gains platform-specific runtime code.
- Never commit Open VSX tokens, Azure credentials, federated tokens, temporary
  passwords, or diagnostic output containing credentials.

## Failures and safe retries

### Hosted agent remains queued

Microsoft-hosted agents can remain at `Acquiring an agent from the cloud` for a
while. Wait for the queued run instead of starting duplicates. Duplicate runs
can race to publish the same immutable version.

### GitHub Release or Open VSX fails

The GitHub Release and Open VSX are separate jobs after the shared build. Rerun
only failed jobs when possible. Before retrying a publication, check whether the
version or GitHub Release was created despite a late failure; registries reject
an already-published version.

If Open VSX authentication fails, verify the protected `release` environment,
the `OVSX_PAT` secret, token expiry, namespace ownership, and account agreement.

### Marketplace returns `InvalidAccessException`

Confirm all of the following:

- the Azure Pipeline is using `abap-language-basics-marketplace`;
- its workload-identity federated credential still matches the service
  connection issuer and subject;
- the service principal still has an Azure DevOps profile;
- the profile resource ID, not an Azure identity ID, is a Contributor on the
  `1meracle1` Marketplace publisher.

After correcting publisher membership, rerun the failed Azure job. The current
pipeline has one job, so its validation and packaging steps run again before
publication.

### Marketplace publish succeeds but the item URL returns 404

New uploads can remain in the publisher portal with status `Verifying` for a few
minutes. During that interval the pipeline is successful and the extension can
be marked Public, while the public item URL still returns 404. Wait for
verification to finish and confirm that the public page shows the expected
version before treating the release as complete.

### A retry says the version already exists

Do not delete or move the release tag to force another upload. Confirm the
registry listing first. If the artifact is already public, record the run as a
late or partially reported success. If its contents are wrong, increment the
package version and publish a new release.
